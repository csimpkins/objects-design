CS 2340 Summer 2013 Final

Problem Description

You will write the core application classes for a course management system (like T-Square).

Software Requirements

Definitions, Acronyms, and Abgreviations

  • Course: a defined semester-long treatment of a set of material within a larger subject, .e.g. CS 2340 Objects and Design
  • Class: an offering of a particular course in a particular semester
  • Section: a list of students enrolled in a particular class.
  • Student: you know what a student is.
  • GradebookCategory: a named or tag applied to GradeBookItems, or named group to which GradebookItems can belong
  • GradebookItem: a thing for which a student receives a score.
  • GradingScheme: an algorithm for computing the "average" score and letter grade for a particular student in a particular section of a particular class based on the student's scores on the GradeBookItems for that class.
  • Score: the numeric score, e.g. 85/100, that a student receives for a particular GradeBookItem
  • Letter grade: the grade a student receives -- A, B, C, D, or F -- for a class based on the student's average score

Overall Description

The software will implement the business logic of a gradebook system, namely, the classes that represent (some of) the problem domain entities defined above and the ways they interrelate, as well as classes that define interfaces for storage (data access objects). The software will not implement any user interface or storage functionality.

User Characteristics

The user of this software is Java code implemented by the user interface team and Java/database code implemented by the database team. This software defines the business logic interface that the UI team will code their software to, and this software defines the interface that the database team should provide.

Constraints

  • The software must be implemented in Java using "plain old Java objects" (no frameworks).
  • The Java objects must be in a package called gradebook.model
  • The software must be tested using Junit.
  • The software must use the supplied Ant build file unmodified.
  • Main source code must be in src/main/java.
  • Unit tests should be in src/test/java.
  • The only things in your GitHub repo should be the build.xml file and lib/ directory you get from your initial fork, and the source code you write in src/main/java and src/test/java.

Specific Requirements

Functional Requirements

  • R1: The software shall allow the user to create a Course with a subject, course number, name, and prerequisite courses.
  • R2: The software shall allow the user to create a Class that contains Sections and reports the average score and letter grade for the course.
  • R3: The software shall allow the user to create a Section that belongs to a Class and contains Students and reports the average score and letter grade for the section.
  • R4: The software shall alllow the user to create a Student with a name who can be added to a Section and given GradebookItem scores and which will report a score and letter grade based on a specified GradingScheme.
  • R5: The software shall allow the user to create a GradebookCategory with a name and a weight to be used in calculating a Student's average score from the scores they received on each GradebookItem.
  • R6: The software shall allow the user to create a GradebookItem with a name and a GradebookCategory.
  • R7: The software shall allow the user to create GradingScheme objects that are used by Student objects to calculate scores and letter grades.
  • R8: The software shall provide interfaces for storing instances of each of its business domain objects in a database.

Design Goals

  • Course, Class, Section, and Student object should report their (average) scores and letter grades in a uniform way.
  • GradingScheme objects should be easily definable to calculate Student scores and letter grades based on user-definable criteria such as GradebookCategory weights, dropping lowest-scored GradebookItem in a GradebookCategory from score calculation, using average scores from the other GradebookItems in a GradebookCategory to replace lowest-scored GradebookItem in the GradebookCategory, and so on.
  • Each type of object should be retrievable from some data store, and it should be easy to create different data stores without modifying much of the code in an application.

Grading

  • Checkstyle Report: 40%
  • JUnit test coverage: 40%
  • Design quality: 20%
